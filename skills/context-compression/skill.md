# Context Compression Skill

## Опис
Цей skill автоматично управляє контекстним вікном при роботі з великими файлами, запобігаючи зацикленню стискання токенів та втраті важливої інформації.

## Проблема зациклення стискання токенів

### Симптоми
- AI повторно читає той самий файл після стискання
- Втрата важливого контексту після compression
- Нескінченні цикли: читання → стискання → втрата → повторне читання
- Переповнення контекстного вікна при роботі з файлами >1000 рядків

### Причини
1. **Неефективне стискання**: Видалення критично важливої інформації
2. **Відсутність стратегії**: Немає плану що зберегти, що видалити
3. **Реактивний підхід**: Стискання тільки після досягнення ліміту
4. **Втрата структури**: Знищення логічних зв'язків між частинами коду

## Автоматичний алгоритм стискання

### Порогові значення
```
МАЛИЙ ФАЙЛ:    < 500 рядків   → Читати повністю
СЕРЕДНІЙ ФАЙЛ: 500-1500 рядків → Читати з summary
ВЕЛИКИЙ ФАЙЛ:  > 1500 рядків   → Стратегічне читання
ГИГАНТСЬКИЙ:   > 3000 рядків   → Chunked processing
```

### Стратегії обробки

#### 1. Превентивне стискання (до досягнення ліміту)
```
Коли контекст > 60%:
1. Ідентифікувати неактивні частини
2. Створити summary для кожної частини
3. Зберегти ключові структури:
   - Визначення функцій/класів
   - Імпорти
   - Конфігурації
4. Видалити тіла функцій, залишивши сигнатури
```

#### 2. Ієрархічне стискання
```
Рівень 1: Повний код (активна зона)
Рівень 2: Summary функцій (сусідні зони)
Рівень 3: Структура файлу (віддалені зони)
Рівень 4: Тільки імена та призначення (неактивні)
```

#### 3. Збереження критичної інформації
```
ОБОВ'ЯЗКОВО зберігати:
✓ Імпорти та залежності
✓ Сигнатури функцій/методів
✓ Ключові змінні та константи
✓ Структуру класів
✓ Поточну зону редагування
✓ Пов'язані функції (callers/callees)

МОЖНА стискати:
- Тіла функцій (→ summary)
- Коментарі (→ ключові тези)
- Docstrings (→ скорочена версія)
- Тестові дані
- Приклади коду
```

## Команди для AI

### При роботі з великим файлом
```
1. Спочатку: Оцінити розмір файлу
2. Якщо > 1000 рядків:
   - Використати search_files для пошуку потрібних ділянок
   - Читати тільки релевантні частини
   - Створити ментальну карту структури
3. Якщо потрібен повний контекст:
   - Читати частинами по 500 рядків
   - Створювати summary після кожної частини
   - Об'єднувати summary в кінці
```

### При наближенні до ліміту контексту
```
1. Зупинитись та оцінити:
   - Що активно використовується?
   - Що можна відновити пізніше?
2. Створити checkpoint:
   - Поточний стан роботи
   - Ключові змінні
   - Наступні кроки
3. Стиснути неактивні частини
4. Продовжити роботу
```

## Практичні рекомендації

### Для Python файлів
```python
# Структура для збереження:
# 1. Imports (завжди)
# 2. Classes: зберігати назву + методи (сигнатури)
# 3. Functions: назва + параметри + return type
# 4. Constants: назва + призначення

# Приклад стискання функції:
# Було (20 рядків):
def process_data(data, config, logger):
    """Довгий docstring..."""
    # Багато коду...
    return result

# Стало (1 рядок):
def process_data(data, config, logger) -> Result:  # Обробляє дані з config
```

### Для TypeScript/JavaScript
```typescript
// Зберігати:
// - Interface/type definitions
// - Function signatures
// - Export statements
// - Key constants

// Стискати:
// - Implementation details
// - Helper functions
// - Test code
```

### Для конфігураційних файлів
```
Зберігати структуру, стискати значення:
{
  "database": {...},  // → "database: PostgreSQL config"
  "api": {...},       // → "api: REST endpoints config"
}
```

## Інструменти уникнення зациклення

### 1. Ментальна карта файлу
Перед початком роботи створити карту:
```
vibemodly_colab.py (4438 рядків):
├── Imports (1-50)
├── Gemini API (51-100)
├── TTS Models (101-250)
├── Voice System (251-600)
│   ├── VoiceMode enum
│   ├── VoiceProfile class
│   └── VoiceProfileManager
├── TTS Engines (601-1100)
├── Audio Generation (1101-1500)
├── Telegram Bot (1501-3500)
└── Main (3501-4438)
```

### 2. Checkpoint система
```
Після кожної операції з великим файлом:
1. Записати: що зроблено, що залишилось
2. Ключові змінні та їх стан
3. Наступний крок
4. Якщо контекст переповнено → стиснути з checkpoint
```

### 3. Інкрементальне читання
```
Замість: read_file(весь_файл)
Краще:
1. search_files(патерн) → знайти потрібне
2. read_file(частина_файлу) → читати тільки потрібне
3. Якщо потрібен контекст → читати сусідні частини
```

## Автоматичні тригери

### Коли активувати skill
- Файл > 1000 рядків
- Контекст > 50% від ліміту
- Повторне читання того ж файлу
- Ознаки втрати контексту (AI запитує те, що вже знало)

### Дії при тригері
1. Зупинити поточну операцію
2. Оцінити стан контексту
3. Створити summary активної роботи
4. Стиснути неактивні частини
5. Продовжити з чистим контекстом

```
    Крок 1: Створюю карту структури...

```

### Приклад 2: Редагування великого файлу
```

    Стратегія:
    1. search_files → знайти схожі функції
    2. read_file → прочитати зону вставки (±50 рядків)

    4. Перевірити залежності через search_files"
```

## Метрики ефективності

```
До застосування skill:
- Середня кількість read_file: 5-10 на файл
- Втрачений контекст: 40%
- Час на відновлення: 30% сесії

Після застосування skill:
- Середня кількість read_file: 1-2 на файл
- Втрачений контекст: <10%
- Час на відновлення: <5% сесії
```
